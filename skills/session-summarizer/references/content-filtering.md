# 内容筛选规则

## 核心原则

**业务优先，技术其次**：记录业务进展和决策，过滤纯技术调试。

---

## ✅ 保留的内容

### 1. 业务进展

| 类型 | 示例 | 判断标准 |
|------|------|---------|
| 客户调研 | 发现张洵是北京清格科技创始股东 | 有意外发现/业务洞察 |
| 技能开发 | customer-research Phase 5 功能补齐 | 有明确产出/可复用 |
| 业务决策 | 选择企查查 API 而非天眼查 | 有决策过程和理由 |
| 数据分析 | 分析 5 位高价值客户特征 | 有业务结论 |

### 2. 重要对话

| 类型 | 示例 | 判断标准 |
|------|------|---------|
| 用户指令 | "创建会话摘要技能" | 明确的工作指令 |
| 用户反馈 | "这个发现很有价值" | 表达态度/评价 |
| 用户决策 | "继续第 1 个任务" | 明确的选择 |

### 3. 反思与洞察

| 类型 | 示例 | 判断标准 |
|------|------|---------|
| 方法论 | "元数据分析法节省 99% Token" | 可复用的经验 |
| 经验教训 | "上下文溢出导致任务漂移" | 有警示价值 |
| 业务洞察 | "客户调研可能发现意外关联" | 有指导意义 |

---

## ❌ 过滤的内容

### 1. 技术调试

| 类型 | 示例 | 过滤原因 |
|------|------|---------|
| API 配置 | 设置 Referer、API Key | 一次性操作 |
| 环境搭建 | 安装依赖、配置路径 | 临时性 |
| 错误排查 | "AMap.HeatMap is not a constructor" | 技术细节 |
| 代码调试 | 修改参数、测试函数 | 过程性 |

**例外**：如果技术调试有重大发现或业务影响，则保留结论部分。

### 2. 中间产物

| 类型 | 示例 | 过滤原因 |
|------|------|---------|
| 测试数据 | 选取 5 个客户测试 | 非最终产出 |
| 临时文件 | heatmap_test.html | 过程文件 |
| 草稿版本 | diary-v1.md, diary-v2.md | 被最终版本替代 |

### 3. 闲聊对话

| 类型 | 示例 | 过滤原因 |
|------|------|---------|
| 问候 | "早上好"、"辛苦了" | 无业务价值 |
| 闲聊 | "今天天气不错" | 与业务无关 |
| 确认 | "好的"、"明白了" | 过程性对话 |

---

## 判断流程

```
对每条内容，依次问 3 个问题：

1. 是否有业务价值？
   - 是 → 保留
   - 否 → 继续下一题

2. 是否是技术调试/中间产物？
   - 是 → 过滤
   - 否 → 继续下一题

3. 是否可复用/有警示价值？
   - 是 → 保留（结论部分）
   - 否 → 过滤
```

---

## 特殊情况处理

### 1. 混合内容（业务 + 技术）

**处理**：保留业务部分，过滤技术细节。

**示例**：
```
✅ 保留："customer-research 技能完成 Phase 5 功能，发现张洵曾是创始股东"
❌ 过滤："调用 886 接口，参数为{...}，返回{...}"
```

### 2. 技术调试但有重大发现

**处理**：保留发现，过滤过程。

**示例**：
```
✅ 保留："发现高德热力图失败原因是 HeatMap 构造函数未定义"
❌ 过滤："尝试了 5 个 HTML 文件，每个都报错..."
```

### 3. 不确定是否有价值

**处理**：保留，但标记为"待确认"。

**示例**：
```markdown
### 待确认事项
- 选取 5 位高价值客户测试（可能是中间产物，需确认）
```
