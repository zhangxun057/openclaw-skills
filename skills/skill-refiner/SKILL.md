# Skill Refiner

**技能打磨专家** — 当现有技能需要优化/迭代时调用

---

## 核心定位

**解决什么问题：**
- 技能表现不佳，用户指出问题
- 技能功能缺失，需要补充
- 技能过于复杂，需要简化（奥卡姆剃刀）
- 技能需要添加新功能，但不想破坏现有结构

**不解决什么：**
- 从零创建新技能（用 skill-creator）
- 技能打包分发（用 skill-creator 的打包功能）

---

## 核心原则

### 原则 1：就事论事（最小修改）

**判断标准：**
- ✅ 修改后能解决一类问题（可复用）
- ❌ 修改后只能解决一个用户的特例（过于具体）

**好示例：**
```
用户：技能不会识别"企业个人股东"
AI：增加"企业个人股东"的识别逻辑（通用，可复用）
```

**不好示例：**
```
用户：技能不会识别"张三"
AI：增加"张三"的识别逻辑（过于具体，不可复用）
```

---

### 原则 2：保持初始目标一致性

**判断方法：** 自问"这个修改是否服务于技能的初始目标？"

- ✅ 是 → 直接修改
- ❌ 否 → 建议单独创建新技能

**示例：**
```
初始目标：客户信息调研
用户要求：增加自动发送邮件功能
AI 判断：超出了初始目标（调研），建议创建"客户邮件发送"技能
```

---

### 原则 3：奥卡姆剃刀（如无必要，勿增实体）

**判断标准：**
- ✅ 新增 ≤ 3 节，新增 ≤ 30 行
- ✅ 能把问题解释清楚
- ❌ 新增 > 5 节，或新增 > 50 行 → 需要反思是否有更简单的方案

---

### 原则 4：先确认，后修改

**禁止行为：**
- ❌ 没有用户确认就直接修改 skill
- ❌ 一次性提供 > 3 个修改方案让用户选择
- ❌ 模糊的修改建议（如"优化一下逻辑"）

**必须做到：**
- ✅ 修改前提供"修改方案"（包含：修改位置、原文、修改后、修改理由）
- ✅ 等待用户确认后再执行
- ✅ 修改后执行测试用例验证

---

## 优化流程（6 步）

### Step 1: 确认场景

**询问用户：**
1. 这是个案问题还是共性问题？
2. 使用场景是什么？
3. 期望的改进效果是什么？

---

### Step 2: 分析原因

**分析维度：**
- 是否过于具体？（如出现具体公司名/人名，应替换为 [公司名称]）
- 是否偏离目标？（修改是否服务于初始目标）
- 是否违反奥卡姆剃刀？（能否用更简单的方案）

---

### Step 3: 提供修改方案

**必须包含：**
1. Skill 目录结构（修改后）
2. 修改位置定位（文件 + 章节 + 原文）
3. 修改内容（精确到每一行）
4. 修改理由（对应哪个原则）

**格式要求：** 使用表格对照或 diff 格式

---

### Step 4: 备份

**执行前备份：**
```bash
python scripts/backup_skill.py <skill-path>
```

**备份规则：**
- 最多保留 5 个历史版本
- 最新版本保留最近修改的备份

---

### Step 5: 执行修改

**修改方式：**
- 使用精确的编辑工具（如 edit）
- 严格按照确认的方案修改
- 新增内容 ≤ 50 行

---

### Step 6: 测试用例验证

**测试用例设计原则：**
- 使用"类客户"而非"真实客户"
- 包含正向测试和回归测试
- 记录测试结果

**测试用例模板：**
```markdown
### 测试用例：企业个人股东识别

**测试输入：** 包含"企业个人股东"类型的股东
**测试点：** AI 是否识别并解释该类型股东
**示例：** 张三是 XX 企业的个人股东，用户未选择
**预期：** AI 识别并解释
```

---

## 修改方案格式（模板）

### 1. Skill 目录结构（修改后）

```
skill-name/
├── SKILL.md
│   ├── ...（原有内容）
│   ├── Step 6: Iterate
│   ├── Step 7: 测试用例验证  ← 新增章节
│   └── Step 8: 版本编号  ← 新增章节
├── scripts/
│   ├── init_skill.py
│   └── package_skill.py
└── references/
```

---

### 2. 修改位置定位

**文件名：** `SKILL.md`
**位置：** "Step 6: Iterate"章节之后，文件末尾之前
**原文内容：**
```markdown
（原文前 3 行）
...
（原文后 3 行）
```

---

### 3. 修改内容（精确表述）

**操作：** 插入/删除/修改

**插入位置：** 在"XXX"之后，"YYY"之前

**插入内容：**
```markdown
（完整内容，不超过 50 行）
```

---

### 4. 修改理由

**对应原则：**
- ✅ 就事论事：修改后能解决一类问题
- ✅ 保持目标一致性：聚焦于"XXX"，不偏离
- ✅ 奥卡姆剃刀：只增加 1 节（13 行），没有更简单的方案
- ✅ 先确认后修改：等待用户确认方案后再执行

---

## 测试用例设计指南

### 设计原则

**测试用例应该"类客户"，而非"真实客户"**

❌ 错误示例（绑定具体公司）：
```markdown
使用"贵州茅台酒股份有限公司"测试：
- 验证是否能识别"白酒行业"的股东
```

✅ 正确示例（类客户）：
```markdown
测试用例类型：企业个人股东识别行业
测试输入：包含"企业个人股东"类型的股东
测试点：AI 是否识别并解释该类型股东
示例：张三是 XX 企业的个人股东，用户未选择
```

---

### 测试用例分类

| 类型 | 描述 | 测试点 |
|------|------|------|
| 企业个人股东 | 股东列表包含"企业个人股东" | 是否识别该类型股东 |
| 历史股东变更 | 变更记录中有股东进出 | 是否识别历史股东 |
| 有负面信息 | 存在被执行/失信记录 | 是否提示风险 |
| 有对外投资 | 投资了其他企业/专利/商标 | 是否反映企业实力 |
| 有融资/投资 | 存在融资/投资/并购事件 | 是否反映企业资本情况 |

---

### 测试流程

```
修改方案确认
    ↓
AI 生成"测试用例执行要求"
    ↓
用户确认是否有不符合的场景（如特定行业）
    ↓
执行测试（用户或 AI 执行）
    ↓
记录测试结果（通过/失败）
    ↓
全部通过 → 完成
    ↓
有失败 → 记录失败原因，重新修改方案
```

## 版本管理规范

### 备份规范

**每次修改前自动备份：**
```bash
cp SKILL.md SKILL.md.backup.YYYYMMDD-HHMMSS
```

**备份文件管理：**
- 最多保留 5 个历史版本
- 最新版本保留最近修改的备份

---

### 版本号规则

| 版本号 | 说明 | 适用场景 |
|--------|------|---------|
| 2.0.0 | 主版本 | 重新定义初始目标或重构 |
| 1.2.0 | 次版本 | 增加新功能或大幅优化 |
| 1.1.1 | 修订号 | bug 修复/错别字/小调整 |

---

### 修改日志

**在 SKILL.md 末尾添加：**

```markdown
---

## 修改记录

| 版本 | 日期 | 修改内容 | 测试用例 |
|------|------|---------|---------|
| 1.0.0 | 2026-04-03 | 初始版本 | - |
```

## 使用示例

### 示例 1：用户要求优化 customer-research 技能

**用户输入：**
```
这个 skill 用得不好，读取数据后不会主动解释哪些信息需要重点验证
应该主动提示哪些是高风险信息，哪些信息需要人工核实
```

**AI 响应流程：**

1. **确认场景：** "您希望 skill 自动识别高风险信息并提示，还是只在发现高风险时才提示？"
2. **分析原因：** 确认是功能缺失（缺少风险识别逻辑）
3. **提供方案：** "增加'风险识别原则'章节，13 行"
4. **用户确认：** 用户确认方案和测试用例
5. **执行备份：** SKILL.md.backup.20260403-230600
6. **执行修改：** 精确修改
7. **执行测试：** 测试风险识别功能
8. **版本编号：** 1.0.0 → 1.1.0
9. **记录日志：** 添加修改记录

---

## 禁止行为

**AI 在迭代优化中禁止：**

1. ❌ 诱导用户直接修改 skill
2. ❌ 一次性提供 > 3 个修改方案
3. ❌ 把简单的需求复杂化（写成通用 skill）
4. ❌ 为了"完整"增加不必要的功能
5. ❌ 跳过测试直接修改
6. ❌ 没有修改方案就直接执行

---

## 总结

**核心要点：**
1. **流程：** 用户提需求 → AI 确认场景 → 提供方案 → 用户确认 → 精确修改
2. **原则：** 就事论事、保持目标一致、奥卡姆剃刀
3. **格式：** 目录结构 + 精确表述（增删改）
4. **禁止：** 跳过确认、超过 3 个方案、复杂化

**最终目标：** 让用户对 skill 的修改拥有完全的控制权，AI 只是"执行者"而非"决策者"


```bash
mkdir -p ~/.openclaw/skill-logs/skill-refiner
```
