# 测试用例模板

## 测试用例格式

```json
{
  "name": "测试用例名称",
  "type": "测试类型",
  "input": "测试输入",
  "expected": "预期结果",
  "actual": "实际结果（执行后填写）",
  "result": "pass/fail（执行后填写）"
}
```

---

## 测试用例分类

### 1. 功能测试

**目的：** 验证修改的功能是否正常工作

**示例：**
```json
{
  "name": "企业个人股东识别",
  "type": "功能测试",
  "input": "包含'企业个人股东'类型的股东信息",
  "expected": "AI 识别并解释该类型股东的含义和风险",
  "actual": "",
  "result": ""
}
```

---

### 2. 回归测试

**目的：** 验证修改没有影响现有功能

**示例：**
```json
{
  "name": "历史股东查询（回归）",
  "type": "回归测试",
  "input": "查询历史股东变更记录",
  "expected": "正常返回历史股东列表，不受新增功能影响",
  "actual": "",
  "result": ""
}
```

---

### 3. 边界测试

**目的：** 验证极端情况下的表现

**示例：**
```json
{
  "name": "无股东信息",
  "type": "边界测试",
  "input": "企业没有股东信息",
  "expected": "AI 提示'未查询到股东信息'，不报错",
  "actual": "",
  "result": ""
}
```

---

### 4. 错误处理测试

**目的：** 验证错误情况的处理

**示例：**
```json
{
  "name": "API 调用失败",
  "type": "错误处理",
  "input": "模拟 API 调用失败",
  "expected": "AI 提示'查询失败，请稍后重试'，不崩溃",
  "actual": "",
  "result": ""
}
```

---

## 测试用例设计指南

### 设计原则

1. **类客户而非真实客户**
   - ❌ "贵州茅台酒股份有限公司"
   - ✅ "XX 企业"

2. **覆盖主要场景**
   - 正常流程
   - 异常流程
   - 边界情况

3. **可重复执行**
   - 测试输入明确
   - 预期结果清晰
   - 可自动化或手动验证

## 测试执行流程

```
修改完成
    ↓
AI 生成测试用例要求
    ↓
用户确认是否有不符合的场景
    ↓
执行测试（用户或 AI 执行）
    ↓
记录结果（pass/fail）
    ↓
全部通过 → 完成
    ↓
有失败 → 记录失败原因，重新修改
```
