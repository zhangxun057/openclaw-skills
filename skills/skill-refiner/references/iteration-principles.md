# Skill 迭代原则

## 核心原则

### 1. 就事论事（最小修改）

**目标：** 修改后能解决一类问题，而非一个特例

**判断标准：**
- ✅ 修改后可以复用到其他类似场景
- ❌ 修改后只能解决当前用户的特定问题

**示例对比：**

❌ 过于具体：
```
用户：技能不会识别"张三"
AI：增加"张三"的识别逻辑
```

✅ 就事论事：
```
用户：技能不会识别"企业个人股东"
AI：增加"企业个人股东"的识别逻辑（通用规则）
```

---

### 2. 保持初始目标一致性

**目标：** 修改服务于技能的初始目标，不偏离

**判断方法：** 自问"这个修改是否服务于技能的初始目标？"

- ✅ 是 → 直接修改
- ❌ 否 → 建议单独创建新技能

**示例：**
```
skill: customer-research（客户背景调研）
初始目标：收集和整理客户企业信息

用户要求：增加自动发送邮件功能
AI 判断：超出初始目标（调研 → 邮件发送）
建议：创建"customer-email-sender"技能
```

---

### 3. 奥卡姆剃刀（如无必要，勿增实体）

**目标：** 用最简单的方案解决问题

**判断标准：**
- ✅ 新增 ≤ 3 节，新增 ≤ 30 行
- ✅ 能把问题解释清楚
- ❌ 新增 > 5 节 或 新增 > 50 行 → 反思是否有更简单的方案

**示例对比：**

❌ 过于复杂：
```
新增 5 个章节，200 行代码
增加 3 个新脚本，2 个配置文件
```

✅ 简单方案：
```
新增 1 个章节，13 行说明
修改 1 个现有函数，5 行代码
```

---

### 4. 先确认，后修改

**目标：** 用户拥有完全的控制权，AI 只是执行者

**禁止行为：**
- ❌ 没有用户确认就直接修改 skill
- ❌ 一次性提供 > 3 个修改方案
- ❌ 模糊的修改建议（如"优化一下逻辑"）

**必须做到：**
- ✅ 修改前提供"修改方案"（包含：修改位置、原文、修改后、修改理由）
- ✅ 等待用户确认后再执行
- ✅ 修改后执行测试用例验证

---

## 修改方案格式

### 标准模板

```markdown
## 修改方案

### 1. Skill 目录结构（修改后）

```
skill-name/
├── SKILL.md
│   ├── ...（原有内容）
│   ├── 新增章节  ← 标注新增
│   └── 修改章节  ← 标注修改
├── scripts/
└── references/
```

### 2. 修改位置定位

**文件名：** `SKILL.md`
**位置：** "XXX"章节之后
**原文内容：**
```markdown
（原文前 3 行）
...
（原文后 3 行）
```

### 3. 修改内容（精确表述）

**操作：** 插入/删除/修改

**插入位置：** 在"XXX"之后，"YYY"之前

**插入内容：**
```markdown
（完整内容，不超过 50 行）
```

### 4. 修改理由

**对应原则：**
- ✅ 就事论事：修改后能解决一类问题
- ✅ 保持目标一致性：聚焦于"XXX"，不偏离
- ✅ 奥卡姆剃刀：只增加 1 节（13 行），没有更简单的方案
- ✅ 先确认后修改：等待用户确认方案后再执行
```

## 测试用例设计

### 设计原则

**测试用例应该"类客户"，而非"真实客户"**

❌ 错误示例（绑定具体公司）：
```markdown
使用"贵州茅台酒股份有限公司"测试：
- 验证是否能识别"白酒行业"的股东
```

✅ 正确示例（类客户）：
```markdown
测试用例类型：企业个人股东识别行业
测试输入：包含"企业个人股东"类型的股东
测试点：AI 是否识别并解释该类型股东
示例：张三是 XX 企业的个人股东，用户未选择
```

---

### 测试用例分类

| 类型 | 描述 | 测试点 |
|------|------|------|
| 企业个人股东 | 股东列表包含"企业个人股东" | 是否识别该类型股东 |
| 历史股东变更 | 变更记录中有股东进出 | 是否识别历史股东 |
| 有负面信息 | 存在被执行/失信记录 | 是否提示风险 |
| 有对外投资 | 投资了其他企业/专利/商标 | 是否反映企业实力 |
| 有融资/投资 | 存在融资/投资/并购事件 | 是否反映企业资本情况 |

## 版本管理规范

### 版本号格式

`Major.Minor.Patch`

| 版本类型 | 格式 | 适用场景 |
|---------|------|---------|
| 主版本 | X.0.0 | 重新定义初始目标或重构 |
| 次版本 | 1.X.0 | 增加新功能/大幅优化现有功能 |
| 修订号 | 1.1.X | bug 修复/错别字/小调整 |

---

### 备份规范

**每次修改前自动备份：**
```bash
cp SKILL.md SKILL.md.backup.YYYYMMDD-HHMMSS
```

**备份文件管理：**
- 最多保留 5 个历史版本
- 最新版本保留最近修改的备份

---

### 修改日志

**在 SKILL.md 末尾添加：**

```markdown
---

## 修改记录

| 版本 | 日期 | 修改内容 | 测试用例 |
|------|------|---------|---------|
| 1.1.0 | 2026-04-03 | 增加 Step 7/8 和迭代原则总结 | 技能迭代测试 |
| 1.0.0 | 2026-04-01 | 初始版本 | - |
```

## 决策树

```
用户提出优化需求
    ↓
AI 分析：是个案还是共性？
    ↓
是个案 → 提供临时解决方案，不修改 skill
    ↓
是共性 → 提供修改方案（最多 3 个）
    ↓
用户确认方案
    ↓
执行备份 → SKILL.md.backup.YYYYMMDD-HHMMSS
    ↓
执行修改（精确到行）
    ↓
执行测试用例
    ↓
测试通过 → 更新版本号，记录日志
    ↓
测试失败 → 回滚到备份，重新设计方案
```

---

## 总结

**四个核心原则：**
1. **就事论事** - 解决一类问题，而非一个特例
2. **保持目标一致** - 不偏离技能初始目标
3. **奥卡姆剃刀** - 用最简单的方案
4. **先确认后修改** - 用户拥有完全控制权

**修改流程：**
确认场景 → 分析原因 → 提供方案 → 用户确认 → 备份 → 修改 → 测试 → 版本编号 → 记录日志

**最终目标：** 让用户对 skill 的修改拥有完全的控制权，AI 只是"执行者"而非"决策者"
