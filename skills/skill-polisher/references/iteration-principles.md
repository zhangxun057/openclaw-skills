# 迭代原则详解

## 1. 奥卡姆剃刀原理

**核心：** 如无必要，勿增实体

### 检查清单

- [ ] 这次修改是否真的必要？
- [ ] 是否有更简单的实现方式？
- [ ] 是否删除了冗余内容而非只添加？
- [ ] 修改后技能是否更简洁？

### 量化标准

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单次修改处数 | ≤3 处 | 避免大改动 |
| 单次修改行数 | ≤50 行 | 保持可控 |
| 技能总行数 | ≤500 行 | 避免臃肿 |
| 参考文件数 | ≤5 个 | 避免分散 |

### 违规案例

❌ **过度设计：**
```markdown
## 异常处理
1. 网络异常 → 重试 3 次，间隔 1s/2s/4s
2. API 异常 → 记录日志，返回错误码
3. 数据异常 → 验证 schema，抛出异常
4. 权限异常 → 检查 token，刷新重试
5. 超时异常 → 动态调整 timeout
```

✅ **简化后：**
```markdown
## 异常处理
失败时重试 1 次，仍失败则报错并记录日志。
```

## 2. 避免过拟合

**核心：** 泛化为通用规则，不硬编码个例

### 检查清单

- [ ] 是否硬编码了特定数据（如企业名、人名）？
- [ ] 替换为"[企业名称]"后是否仍成立？
- [ ] 这个规则是否适用于其他场景？
- [ ] 是否把个例当成了通用规则？

### 违规案例

❌ **过拟合：**
```python
if company_name == "北京清格科技有限公司":
    # 特殊处理
    search_keywords = ["智慧政务", "ToG", "高新技术企业"]
```

✅ **泛化后：**
```python
# 根据企业类型动态生成搜索词
if "科技" in company_name:
    search_keywords = ["高新技术", "专利技术", "研发投入"]
elif "政务" in company_name:
    search_keywords = ["ToG", "政府项目", "招投标"]
```

## 3. 目标一致性

**核心：** 服务于技能初始目标，不偏离核心用途

### 检查清单

- [ ] 这个功能是否服务于技能的初始目标？
- [ ] 是否添加了"可能有用的"边缘功能？
- [ ] 是否偏离了技能的核心用途？
- [ ] 用户触发这个技能时期望的是什么？

### 违规案例

❌ **偏离目标：**
```markdown
# customer-research 技能
...
## 附加功能
- 生成 PPT 报告
- 发送邮件给客户
- 创建飞书任务
- 同步到 CRM 系统
```

✅ **聚焦目标：**
```markdown
# customer-research 技能
**核心目标：** 搜索企业背景信息，输出结构化调研报告

**功能边界：**
- ✅ 企业信息搜索（web_search + API）
- ✅ 调研报告生成
- ❌ PPT 生成（用其他技能）
- ❌ 邮件发送（用其他技能）
```

## 应用场景

### 何时使用这些原则

| 场景 | 使用原则 |
|------|---------|
| 用户反馈"太复杂" | 奥卡姆剃刀 |
| 技能在个例上有效，其他场景失败 | 避免过拟合 |
| 技能功能越来越多，但核心功能变弱 | 目标一致性 |
| 修改前评估方案 | 三原则全部检查 |
| 修改后验证质量 | 三原则全部检查 |

---

## 决策流程

```
用户反馈问题
    ↓
Step 1: 是个例还是共性？
    ↓ 共性
Step 2: 违反哪个原则？
    ├─ 太复杂 → 奥卡姆剃刀
    ├─ 不通用 → 避免过拟合
    └─ 偏离目标 → 目标一致性
    ↓
Step 3: 提出修改方案（≤3 条）
    ↓
Step 4: 用户确认
    ↓
Step 5: 执行修改
    ↓
Step 6: 测试验证
```
