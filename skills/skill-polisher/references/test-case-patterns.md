# 测试案例类型库

## 测试案例设计原则

1. **覆盖核心场景** - 测试技能的主要用途
2. **边界条件** - 测试极端情况
3. **异常处理** - 测试错误场景
4. **可重复执行** - 每次测试结果一致

---

## 测试案例模板

```markdown
### 案例名称

**输入：** 用户请求/触发条件

**预期行为：** 技能应该做什么

**预期输出：** 应该产生什么结果

**验证方法：** 如何确认测试通过
```

---

## 常见测试类型

### 1. 基本功能测试

验证技能的核心功能是否正常工作。

**示例（customer-research）：**
```
案例：企业信息搜索

输入："调研一下北京清格科技有限公司"

预期行为：
- 调用 web_search 搜索企业信息
- 调用企查查 API 获取详细数据
- 生成结构化报告

预期输出：
- 报告文件：workspace/storage/北京清格科技/99-背景调研/xxx.md
- 报告内容：企业规模、行业地位、主营业务

验证方法：
- 检查报告文件是否存在
- 检查报告内容是否包含关键信息
```

---

### 2. 边界条件测试

测试极端或边界情况。

**示例（customer-research）：**
```
案例：超短企业名称

输入："调研一下腾讯"

预期问题：
- 名称太短，可能匹配多个公司
- 需要确认具体是哪一家

预期行为：
- 列出匹配的公司列表
- 请求用户确认

验证方法：
- 检查是否正确识别歧义
- 检查是否有确认话术
```

---

### 3. 异常处理测试

测试错误场景的处理。

**示例（customer-research）：**
```
案例：API Key 失效

输入：任何调研请求

模拟条件：
- 企查查 API Key 过期或无效

预期行为：
- 检测到 API 错误
- 降级到 web_search
- 告知用户 API 不可用

预期输出：
- 仍然输出报告（基于 web_search）
- 包含提示："企查查 API 不可用，仅使用公开搜索"

验证方法：
- 使用无效 API Key 测试
- 检查是否有降级处理
```

---

### 4. 多公司确认测试

测试同名公司的处理。

**示例（customer-research）：**
```
案例：多个同名公司

输入："调研一下清格科技"

预期行为：
- 搜索到多个公司（北京/上海/深圳）
- 列出所有匹配项
- 请求用户确认

预期输出：
```
搜索到 3 家"清格科技"：
1. 北京清格科技有限公司 - 智慧政务 - 2019 年成立
2. 上海清格科技有限公司 - 环保科技 - 2020 年成立
3. 深圳清格科技有限公司 - 信息技术 - 2021 年成立

请问是哪一家？
```

验证方法：
- 搜索常见企业名称
- 检查是否有确认话术
```

---

### 5. 输出格式测试

测试报告格式是否正确。

**示例（customer-research）：**
```
案例：报告格式验证

输入："调研一下 XXX 公司"

预期输出格式：
```markdown
# XXX 公司背景调研报告

## 一、企业概况
- 成立时间
- 注册资本
- 员工规模

## 二、行业地位
- 市场份额
- 竞争对手
- 核心优势

## 三、主营业务
- 产品/服务
- 目标客户
- 收入模式

## 四、调研建议
- 拜访建议
- 合作机会
```

验证方法：
- 检查 Markdown 格式
- 检查章节完整性
- 检查存储路径正确
```

---

### 6. 性能测试

测试执行时间和资源消耗。

**示例（customer-research）：**
```
案例：调研耗时

输入："调研一下 XXX 公司"

预期性能：
- 总耗时：<2 分钟
- API 调用次数：≤5 次
- Token 消耗：<50K

验证方法：
- 记录开始/结束时间
- 统计 API 调用次数
- 检查 Token 使用量
```

## 测试执行清单

执行技能测试时，按以下顺序：

- [ ] 1. 基本功能测试（核心场景）
- [ ] 2. 边界条件测试（极端情况）
- [ ] 3. 异常处理测试（错误场景）
- [ ] 4. 输出格式测试（格式验证）
- [ ] 5. 性能测试（可选）
